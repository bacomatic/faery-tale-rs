## Bug #133: World items always render on top of player - [#133](https://github.com/bacomatic/faery-tale-rs/issues/133)

**Filed:** 2026-04-01
**Status:** investigating

### Description

World objects (items on the ground such as chests, keys, money bags) always render on top of the player sprite, regardless of relative Y positions. If the player walks south past an item, the item should appear behind the player (player drawn on top), but instead the item visually floats above the hero.

### Investigation

- **Affected files:**
  - `src/game/gameplay_scene.rs` (lines 3298–3322 render pass)

- **Root cause:** Confirmed — incorrect render order.

  In the original game (`fmain.c:2780–3184`), **all** animated entities — the hero, enemies, setfig NPCs, and world objects — are added to a single `anim_list[]` array, Y-sorted via bubble sort (ascending `abs_y`), and rendered in that order. Sprites lower on screen (higher Y) are drawn on top of sprites higher on screen (lower Y). Tile-based depth masking (`mask_blit`) is applied per-sprite afterward.

  In the Rust port, world objects are blitted in a **separate pass after all actors** (lines 3309–3322), unconditionally overwriting any hero/NPC pixels already in the framebuffer. There is no Y-sorting between sprites and objects.

  Render order in the Rust port (current, broken):
  1. Tile map background (`mr.compose`)
  2. Hero sprite → Enemy NPCs → SetFig NPCs (via `blit_actors_to_framebuf`)
  3. **World objects** ← always last, always on top
  4. Tile-based sprite-depth masking (`apply_sprite_mask`)

  Correct order (original game):
  1. Tile map background
  2. All sprites + objects Y-sorted together, drawn in ascending Y order
  3. Per-sprite tile masking

- **Reproduction path:** Place any ground item (e.g., the starting chest at 19298,16128 in Region 3) and walk the hero south past it. The item renders on top of the hero even though the hero is in front of it.

- **Evidence:** `gameplay_scene.rs` lines 3309–3322 blit world objects after `blit_actors_to_framebuf()` returns at line 3307. No Y-interleaving exists between actors and objects.

## Bug #134: Lightlevel never reaches full brightness - [#134](https://github.com/bacomatic/faery-tale-rs/issues/134)

**Status:** investigating

### Description

When the game starts, there's a moment of full brightness (the base palette displayed raw before the first `compute_current_palette` call), then the scene dims because `lightlevel` is adjusted for time of day.   let g_pct = ((ll - 61) * 100 / 300).clamp(0, 100) as i16;
  ```rust
  let b_pct = ((ll - 62) * 100 / 300).clamp(0, 100) as i16;
  ```

  This incorrectly normalizes the lightlevel range [0–300] to [0–100], but the original never does this — `fade_page` handles its own clamping.

  **Effect at key lightlevels:**


## Bug #137: SetFig Y-offset double-subtraction renders NPCs 18px too high - [#137](https://github.com/bacomatic/faery-tale-rs/issues/137)

**Filed:** 2026-04-01
**Status:** planned

### Description

The barkeeper in the tavern (region 8, indoors, near 2969,33982) is not appearing. Investigation reveals all SetFig NPCs (wizard, priest, guards, king, bartender, witch, ranger, beggar, etc.) are rendered 18 pixels too high due to a double-subtraction of the Y-offset. In indoor buildings this pushes NPCs into ceiling tile areas where sprite-depth masking hides them behind wall tiles, making them invisible.

### Investigation

- **Affected files:**
  - `src/game/gameplay_scene.rs` — SetFig rendering branch (lines 3436–3461)

- **Root cause:** Confirmed — double Y-offset subtraction.

  In the original game (`fmain2.c:1589-1630`), set_objects calculates:
  ```c
  ystart = list->yc - map_y - 8;   // base object offset
  ystart -= 18;                     // setfig adjustment
  // Total: yc - map_y - 26
  ```

  In the Rust port (`gameplay_scene.rs:3446-3447`):
  ```rust
  let (rel_x, rel_y_base) = Self::actor_rel_pos(obj.x, obj.y, map_x, map_y);
  // actor_rel_pos uses offsets (-8, -26) — the hero's Y offset, not the object base offset of -8
  let rel_y = rel_y_base - 18;
  // Total: abs_y - map_y - 26 - 18 = abs_y - map_y - 44
  ```

  `actor_rel_pos` was designed for the hero with offsets `(-8, -26)`. The original hero rendering uses `rel_y = hero_y - map_y - 26`. It happens that the SetFig final offset is also -26 (via -8 - 18), so using `actor_rel_pos` for SetFigs is correct **without** the additional `-18`. The code applies both the hero's `-26` AND the SetFig's `-18`, resulting in `-44` instead of the correct `-26`.

  **Effect:** All SetFig NPCs are positioned 18 pixels too high. For the barkeeper at (2966, 33964) with the camera near hero (2969, 33982):
  - Correct position: rel_y ≈ 26 (sprite at Y=26 to Y=58, clearly visible)
  - Buggy position: rel_y ≈ 8 (sprite at Y=8 to Y=40, overlapping ceiling area)

  In indoor buildings, ceiling/wall tiles at the top of the viewport have sprite-depth mask data. The `apply_sprite_mask` function re-stamps tile pixels over the sprite area, hiding the barkeeper behind the ceiling.

- **Reproduction path:** Enter any indoor tavern in region 8 and look for the barkeeper. All four barkeepers (at 9631/38953, 10191/38953, 10649/38953, 2966/33964) will be missing or partially hidden by ceiling tiles.

- **Evidence:**
  - Original `fmain2.c:1589`: `ystart = list->yc - map_y - 8;`
  - Original `fmain2.c:1630`: `ystart -= 18;` (total: -26)
  - Rust `gameplay_scene.rs:2484`: `actor_rel_pos_offset(..., -8, -26)` (hero Y offset)
  - Rust `gameplay_scene.rs:3447`: `let rel_y = rel_y_base - 18;` (extra -18 → total -44)

- **Related:** Follow-up to #136 (SetFig data pipeline). The ob_stat 3 loading fix was applied but this rendering offset bug was introduced.

### Fix Design

**Approach A — Remove the extra `- 18`** (approved)

In `gameplay_scene.rs` line 3447, change:
```rust
let rel_y = rel_y_base - 18;
```
to:
```rust
// Original uses ystart = yc - map_y - 8, then ystart -= 18 (total: -26).
// actor_rel_pos already applies a Y offset of -26, so no further adjustment needed.
let rel_y = rel_y_base;
```

Add a comment explaining why the code differs from the original's two-step subtraction: `actor_rel_pos` combines what the original does in two steps (-8 base, -18 setfig adjustment) into a single -26 offset.

## Bug #138: Sprite-depth masking applied as batch overwrites hero pixels near setfigs - [#138](https://github.com/bacomatic/faery-tale-rs/issues/138)

**Filed:** 2026-04-01
**Status:** investigating

### Description

Standing in front of a setfig causes the terrain to render over the player. The hero sprite gets partially or fully overwritten by tile pixels when near a setfig NPC, even though the hero should be drawn on top (higher Y = closer to camera).

### Investigation

- **Affected files:**
  - `src/game/gameplay_scene.rs` — lines 3466–3469 (masking pass)
  - `src/game/sprite_mask.rs` — `apply_sprite_mask()`

- **Root cause:** Confirmed — two-pass render/mask architecture violates the original per-sprite masking order.

  In the original game (`fmain.c:2887–3197`), each sprite is fully processed individually in Y-sorted order:
  1. `clear_blit(bmask_mem)` — clear mask buffer
  2. `maskit()` loop (lines 3136–3184) — build per-tile mask based on terrain
  3. `mask_blit()` — apply mask (terrain shows through masked sprite regions)
  4. `shape_blit()` — draw sprite with mask

  Each sprite is **completely drawn and masked** before the next sprite is processed. When sprite B (hero, higher Y) is drawn after sprite A (setfig, lower Y), B's pixels are never affected by A's masking — A's masking was already applied before B was drawn.

  In the Rust port, rendering is split into two passes:
  - **Pass 1** (lines 3323–3464): Blit all sprites to framebuf in Y-sorted order
  - **Pass 2** (lines 3466–3469): Apply `apply_sprite_mask()` for all sprites

  When Pass 2 runs masking for sprite A (the setfig):
  - It iterates tiles overlapping A's bounding box
  - For tiles with non-zero mask types, it re-stamps tile pixels over A's area in the framebuf
  - But **sprite B (hero) was already drawn in that area** during Pass 1
  - So tile pixels overwrite the hero's pixels → terrain renders over the player

  When Pass 2 then runs masking for sprite B (the hero):
  - Additional tile pixels may be stamped over the hero's area
  - The hero is now partially or fully hidden behind terrain

- **Reproduction path:** Stand the hero south of (below, higher Y than) any visible setfig NPC. The hero sprite will have terrain tiles rendered over it in the overlap region.

- **Evidence:**
  - Original `fmain.c:2926-2928`: `clear_blit(bmask_mem)` per-sprite, before masking
  - Original `fmain.c:3181`: `maskit()` called per-sprite
  - Original `fmain.c:3187-3188`: `mask_blit(); shape_blit();` per-sprite, before next sprite
  - Rust `gameplay_scene.rs:3466-3469`: masking applied in a separate loop after all sprites are drawn

- **Additional issue:** `_actor_idx` is hardcoded to `0` in all `apply_sprite_mask()` calls (line 3468), making `is_actor_1` always false. This breaks mask type 3 (bridge sector) for the raft (actor index 1). Minor but worth fixing alongside.
