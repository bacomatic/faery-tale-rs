## Bug #133: World items always render on top of player - [#133](https://github.com/bacomatic/faery-tale-rs/issues/133)

**Filed:** 2026-04-01
**Status:** investigating

### Description

World objects (items on the ground such as chests, keys, money bags) always render on top of the player sprite, regardless of relative Y positions. If the player walks south past an item, the item should appear behind the player (player drawn on top), but instead the item visually floats above the hero.

### Investigation

- **Affected files:**
  - `src/game/gameplay_scene.rs` (lines 3298–3322 render pass)

- **Root cause:** Confirmed — incorrect render order.

  In the original game (`fmain.c:2780–3184`), **all** animated entities — the hero, enemies, setfig NPCs, and world objects — are added to a single `anim_list[]` array, Y-sorted via bubble sort (ascending `abs_y`), and rendered in that order.   In the Rust port, world objects are blitted in a **separate pass after all actors** (lines 3309–3322), unconditionally overwriting any hero/NPC pixels already in the framebuffer. There is no Y-sorting between sprites and objects.

  Render order in the Rust port (current, broken):
  1. Tile map background (`mr.compose`)
  2. Hero sprite → Enemy NPCs → SetFig NPCs (via `blit_actors_to_framebuf`)
  3. **World objects** ← always last, always on top
  4. Tile-based sprite-depth masking (`apply_sprite_mask`)

  Correct order (original game):
  1. Tile map background
  2. All sprites + objects Y-sorted together, drawn in ascending Y order
  3. Per-sprite tile masking

- **Reproduction path:** Place any ground item (e.g., the starting chest at 19298,16128 in Region 3) and walk the hero south past it. The item renders on top of the hero even though the hero is in front of it.

- **Evidence:** `gameplay_scene.rs` lines 3309–3322 blit world objects after `blit_actors_to_framebuf()` returns at line 3307. No Y-interleaving exists between actors and objects.

### Investigation

- **Root cause:** Confirmed — double Y-offset subtraction.

  In the original game (`fmain2.c:1589-1630`), set_objects calculates:
  ```c
  ystart = list->yc - map_y - 8;   // base object offset
  ystart -= 18;                     // setfig adjustment
  // Total: yc - map_y - 26
  ```

  In the Rust port (`gameplay_scene.rs:3446-3447`):
  ```rust
  let (rel_x, rel_y_base) = Self::actor_rel_pos(obj.x, obj.y, map_x, map_y);
  // actor_rel_pos uses offsets (-8, -26) — the hero's Y offset, not the object base offset of -8
  let rel_y = rel_y_base - 18;
  // Total: abs_y - map_y - 26 - 18 = abs_y - map_y - 44
  ```

  `actor_rel_pos` was designed for the hero with offsets `(-8, -26)`. The original hero rendering uses `rel_y = hero_y - map_y - 26`. The code applies both the hero's `-26` AND the SetFig's `-18`, resulting in `-44` instead of the correct `-26`.

  - Correct position: rel_y ≈ 26 (sprite at Y=26 to Y=58, clearly visible)
  - Buggy position: rel_y ≈ 8 (sprite at Y=8 to Y=40, overlapping ceiling area)

  In indoor buildings, ceiling/wall tiles at the top of the viewport have sprite-depth mask data. The `apply_sprite_mask` function re-stamps tile pixels over the sprite area, hiding the barkeeper behind the ceiling.

- **Reproduction path:** Enter any indoor tavern in region 8 and look for the barkeeper. All four barkeepers (at 9631/38953, 10191/38953, 10649/38953, 2966/33964) will be missing or partially hidden by ceiling tiles.

- **Evidence:**
  - Original `fmain2.c:1589`: `ystart = list->yc - map_y - 8;`
  - Original `fmain2.c:1630`: `ystart -= 18;` (total: -26)
  - Rust `gameplay_scene.rs:2484`: `actor_rel_pos_offset(..., -8, -26)` (hero Y offset)
  - Rust `gameplay_scene.rs:3447`: `let rel_y = rel_y_base - 18;` (extra -18 → total -44)

- **Related:** Follow-up to #136 (SetFig data pipeline). The ob_stat 3 loading fix was applied but this rendering offset bug was introduced.

### Fix Design

**Approach A — Remove the extra `- 18`** (approved)

In `gameplay_scene.rs` line 3447, change:
```rust

```
to:

  In the Rust port, rendering is split into two passes:

  - **Pass 2** (lines 3466–3469): Apply `apply_sprite_mask()` for all sprites

  - It iterates tiles overlapping A's bounding box
  - For tiles with non-zero mask types, it re-stamps tile pixels over A's area in the framebuf
  - But **sprite B (hero) was already drawn in that area** during Pass 1
  - So tile pixels overwrite the hero's pixels → terrain renders over the player

  When Pass 2 then runs masking for sprite B (the hero):
  - Additional tile pixels may be stamped over the hero's area
  - The hero is now partially or fully hidden behind terrain

  - Original `fmain.c:3181`: `maskit()` called per-sprite
  - Original `fmain.c:3187-3188`: `mask_blit(); shape_blit();` per-sprite, before next sprite
  - Rust `gameplay_scene.rs:3466-3469`: masking applied in a separate loop after all sprites are drawn

### Description
When the player walks through bushes, the sprite sinks downward and takes damage as if walking in lava. Bushes should be passable terrain that at most slows movement, not cause submersion or health loss.

### Investigation
- **Affected files:** `src/game/gameplay_scene.rs` (lines 742-752 submerged check, lines 2575/3332 Y-offset, lines 2914-2922 drowning damage), `src/game/collision.rs` (px_to_terrain_type)
- **Root cause:** The Rust port replaces the original's graduated `environ` sinking system with a binary `submerged` flag that triggers immediately on terrain type 2. This collapses multiple distinct terrain behaviors into a single "underwater" state.

  **How it works in the original (`fmain.c:2019-2074`):**
  The sinker loop processes terrain types and sets `k` (which becomes `an->environ`):
  - Type 0 → k=0 (reset, surface)
  - Type 2 → k=2 (shallow — wading depth, just visual)
  - Type 3 → k=5 (brush/deep — slight visual sink, still no damage)
  - Type 4 → gradual increment toward threshold 10; SINK state at 15
  - Type 5 → gradual increment toward threshold 30; full submersion, drowning at 30
  - Type 6 → k=-1 (ice)
  - Type 7 → k=-2 (lava — damage via `fiery_death` flag, not via sinking)

  The rendering uses `environ` as a Y-offset (`ystart += an->environ`, `fmain.c:3038-3051`):
  - `environ > 2` → sprite shifts down by `environ` pixels (gradual sink)
  - `environ > 29` → fully submerged, replaced with splash animation
  - `environ <= 2` → no visual change (shallow wading only)

  Drowning damage (`fmain.c:2142-2146`) only at `environ == 30 && (cycle & 7) == 0`.

  Lava damage (`fmain.c:2133-2140`) is separate: controlled by `fiery_death` global (map bounds check for volcanic region, `fmain.c:1554-1555`), applies `vitality--` when `environ > 2`.

  **How it works in the Rust port (broken):**
  - Line 749: `self.submerged = terrain == 2;` — any terrain type 2 tile triggers binary submersion
  - Lines 2575/3332: `if hero_submerged { rel_y += 8; }` — hardcoded 8px sink instead of gradual environ-based offset
  - Lines 2914-2922: `if self.submerged { ... vitality -= 1 }` — instant drowning damage every ~1s
  - Line 463: `if (2..=5).contains(&terrain) { 1 } else { 2 }` — speed halved on types 2-5 (this part is approximately correct)

  If bushes are painted with terrain type 2 (shallow water/wading) or type 3 (brush), the Rust port immediately sinks the player and applies drowning damage. The original would only set environ=2 (type 2) or environ=5 (type 3), which causes a slight visual dip but NO damage until environ reaches 30 (which only happens by staying on type 4/5 terrain).

  **The entire `environ` graduated sinking system is missing from the Rust port.** The binary submerged flag is a lossy simplification that conflates wading (environ=2, harmless), brush (environ=5, harmless), shallow water (environ gradually→10), and deep water (environ gradually→30, drowning).

- **Evidence:**
  - Original sinker: `fmain.c:2019-2074` — sets environ per terrain type
  - Original Y-offset: `fmain.c:3038-3051` — `ystart += an->environ`
  - Original drowning: `fmain.c:2142-2146` — only at `environ==30`
  - Original lava: `fmain.c:2133-2140` — `fiery_death` flag from map bounds
  - Rust submerged: `gameplay_scene.rs:749` — `terrain == 2` triggers instantly
  - Rust damage: `gameplay_scene.rs:2914-2922` — 1 vit/sec while submerged

### Fix Design
**Approach: Port full `environ` system**

Replace binary `submerged` flag with the original's graduated environ from `fmain.c:2019-2074`:
1. Per-terrain-type environ assignment: type 0→0, type 2→2, type 3→5, type 6→−1, type 7→−2, type 8→−3
2. Gradual increment/decrement for types 4/5 toward thresholds (10 and 30)
3. SINK animation state trigger at environ>15
4. Proportional Y-offset rendering: `ystart += environ` (replaces hardcoded `rel_y += 8`)
5. Full submersion rendering at environ>29 (splash sprite)
6. Drowning damage only at environ==30 && (cycle & 7)==0
7. Separate `fiery_death` lava damage: region bounds check (`fmain.c:1554-1555`), `environ > 2 → vitality--`, `environ > 15 → vitality = 0`, Rose item (stuff[23]) immunity
