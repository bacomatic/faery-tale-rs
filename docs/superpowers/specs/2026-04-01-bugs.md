## Bug #133: World items always render on top of player - [#133](https://github.com/bacomatic/faery-tale-rs/issues/133)

**Filed:** 2026-04-01
**Status:** investigating

### Description

World objects (items on the ground such as chests, keys, money bags) always render on top of the player sprite, regardless of relative Y positions. If the player walks south past an item, the item should appear behind the player (player drawn on top), but instead the item visually floats above the hero.

### Investigation

- **Affected files:**
  - `src/game/gameplay_scene.rs` (lines 3298–3322 render pass)

- **Root cause:** Confirmed — incorrect render order.

  In the original game (`fmain.c:2780–3184`), **all** animated entities — the hero, enemies, setfig NPCs, and world objects — are added to a single `anim_list[]` array, Y-sorted via bubble sort (ascending `abs_y`), and rendered in that order. Sprites lower on screen (higher Y) are drawn on top of sprites higher on screen (lower Y). Tile-based depth masking (`mask_blit`) is applied per-sprite afterward.

  In the Rust port, world objects are blitted in a **separate pass after all actors** (lines 3309–3322), unconditionally overwriting any hero/NPC pixels already in the framebuffer. There is no Y-sorting between sprites and objects.

  Render order in the Rust port (current, broken):
  1. Tile map background (`mr.compose`)
  2. Hero sprite → Enemy NPCs → SetFig NPCs (via `blit_actors_to_framebuf`)
  3. **World objects** ← always last, always on top
  4. Tile-based sprite-depth masking (`apply_sprite_mask`)

  Correct order (original game):
  1. Tile map background
  2. All sprites + objects Y-sorted together, drawn in ascending Y order
  3. Per-sprite tile masking

- **Reproduction path:** Place any ground item (e.g., the starting chest at 19298,16128 in Region 3) and walk the hero south past it. The item renders on top of the hero even though the hero is in front of it.

- **Evidence:** `gameplay_scene.rs` lines 3309–3322 blit world objects after `blit_actors_to_framebuf()` returns at line 3307. No Y-interleaving exists between actors and objects.

## Bug #134: Lightlevel never reaches full brightness - [#134](https://github.com/bacomatic/faery-tale-rs/issues/134)

**Filed:** 2026-04-01
**Status:** investigating

### Description

When the game starts, there's a moment of full brightness (the base palette displayed raw before the first `compute_current_palette` call), then the scene dims because `lightlevel` is adjusted for time of day. Even at noon (ll=300, maximum), the palette never reaches full brightness — it caps around 73%.

### Investigation

- **Affected file:** `src/game/gameplay_scene.rs`, `compute_current_palette()` lines 2371–2375

- **Root cause:** Confirmed — spurious `* 100 / 300` scaling on the channel parameters.

  The original C code (`fmain2.c:day_fade`) passes `(lightlevel - 80, lightlevel - 61, lightlevel - 62)` **directly** to `fade_page()`. Those values are already treated as percentages and clamped inside `fade_page` to [10, 100].

  The Rust code adds an extra `* 100 / 300` scaling:
  ```rust
  let r_pct = ((ll - 80 + ll_boost) * 100 / 300).clamp(0, 100) as i16;
  let g_pct = ((ll - 61) * 100 / 300).clamp(0, 100) as i16;
  let b_pct = ((ll - 62) * 100 / 300).clamp(0, 100) as i16;
  ```

  This incorrectly normalizes the lightlevel range [0–300] to [0–100], but the original never does this — `fade_page` handles its own clamping.

  **Effect at key lightlevels:**

  | ll  | Channel | Original | Rust (buggy) |
  |-----|---------|----------|--------------|
  | 300 | Red     | 220→100  | 73           |
  | 300 | Green   | 239→100  | 79           |
  | 300 | Blue    | 238→100  | 79           |
  | 180 | Red     | 100      | 33           |
  | 120 | Red     | 40       | 13           |

  Full brightness (all channels at 100%) should occur at ll≥180 (documented in RESEARCH.md). With the bug, it **never** occurs.

- **Fix:** Remove the `* 100 / 300` scaling. Pass `(ll - 80 + ll_boost)`, `(ll - 61)`, `(ll - 62)` directly as `i16`, matching the original:
  ```rust
  let r_pct = (ll - 80 + ll_boost) as i16;
  let g_pct = (ll - 61) as i16;
  let b_pct = (ll - 62) as i16;
  ```
  `fade_page()` already clamps to [10, 100] when `limit=true`.

## Bug #135: Fatigue is movement-based instead of purely time-based

**Filed:** 2026-04-01
**Status:** investigating

### Description

Fatigue decrements when the player is stationary (wrong) and increments on movement (also wrong). In the original game, fatigue is purely time-based — it ticks up at the same rate as hunger (every 128 `daynight` ticks) and only decrements when the player sleeps.

### Investigation

- **Affected files:**
  - `src/game/game_state.rs` — `fatigue_step()` and `hunger_fatigue_step()`
  - `src/game/gameplay_scene.rs` — calls `fatigue_step()` on every movement step

- **Root cause:** Two bugs compounding.

  **Bug A:** `hunger_fatigue_step()` (line 395) increments `self.hunger += 1` but is **missing `self.fatigue += 1`**. The original (`fmain.c:2624–2625`) increments both together:
  ```c
  hunger++;
  fatigue++;
  ```

  **Bug B:** An invented `fatigue_step(moved: bool)` method (line 751) does `+1` on movement, `-1` on rest, and triggers forced sleep at `MAX_FATIGUE=200`. This has no counterpart in the original. The original only modifies fatigue in two places:
  1. `fatigue++` every 128 daynight ticks (time-based, alongside hunger)
  2. `fatigue--` per frame while in SLEEP state

  The `fatigue_step()` method is called per-movement-step in `gameplay_scene.rs:754`.

- **Fix:**
  1. Add `self.fatigue += 1` to `hunger_fatigue_step()` (alongside hunger)
  2. Remove `fatigue_step()` entirely from `game_state.rs`
  3. Remove the `fatigue_step()` call and its forced-sleep handler from `gameplay_scene.rs:754–761`
  4. The forced-sleep-from-exhaustion is already handled in `hunger_fatigue_step()` at fatigue>170 (event 12)

## Bug #136: SetFig NPCs not appearing in the world

**Filed:** 2026-04-01
**Status:** investigating

### Description

SetFig NPCs (stationary named characters — wizards, priests, guards, shopkeepers, kings, witches, beggars, etc.) do not appear in the world. The rendering infrastructure exists but receives no data.

### Investigation

- **Affected files:**
  - `src/game/game_state.rs` — `populate_region_objects()` skips ob_stat 3/4
  - `src/game/npc.rs` — `NpcTable::load()` reads ADF cfile data (enemies), not faery.toml setfigs
  - `src/game/gameplay_scene.rs` — setfig render pass (lines 2656–2673) iterates NpcTable which contains no setfigs

- **Root cause:** Missing data pipeline.

  The original game (`fmain2.c:set_objects()`) iterates the same object lists for ALL ob_stat values. When it encounters `ob_stat == 3` (setfig), it:
  1. Looks up `setfig_table[ob_id]` to get cfile_entry and image_base
  2. Loads the sprite sheet if needed
  3. Adds the setfig to `anim_list[]` with `type = SETFIG`, `state = STILL`, `race = ob_id + 0x80`
  4. SetFigs are then Y-sorted and rendered alongside all other sprites

  In the Rust port:
  - `faery.toml` contains 32 setfig entries (ob_stat=3) — **data exists**
  - `populate_region_objects()` explicitly filters to `ob_stat == 1 || ob_stat == 5` — **setfigs skipped**
  - The setfig render pass reads from `NpcTable` which loads from ADF cfile — **wrong data source**
  - No code bridges faery.toml setfig entries to the render pipeline

- **Fix approach:** When populating region objects, entries with ob_stat 3 should be loaded into a setfig-specific list (or the existing WorldObject list with a distinguishing flag). The render pass then draws them using `SETFIG_TABLE[ob_id]` to look up the correct sprite sheet and frame. Key details from the original:
  - SetFig `ob_id` is the setfig_table index (0–13), NOT an obytes sprite frame
  - `race = ob_id + 0x80` encodes the setfig type
  - SetFigs use `cfiles[13–17]` (already loaded at startup)
  - Y-offset: original does `ystart -= 18` for setfigs
