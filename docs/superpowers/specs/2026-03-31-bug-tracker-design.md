# Bug Tracker Skill Design

**Date:** 2026-03-31

## Overview

A `/bug` VS Code slash command that accepts a short description, investigates the bug deeply using codebase exploration, records findings in a growable session-scoped spec file, files a GitHub issue, designs the fix with user approval (brainstorming-style), and transitions to writing-plans.

---

## Architecture

Two files, mirroring the brainstorming pattern:

- **`skills/bug-tracker/SKILL.md`** — full workflow logic; lives in `~/../superpowers/skills/bug-tracker/`
- **`.github/prompts/bug.prompt.md`** — thin VS Code slash command; captures `${input:Bug description}` and instructs the agent to load the bug-tracker skill

The prompt is the doorbell; the skill is the brain.

---

## Session Spec File

**Path:** `docs/superpowers/specs/YYYY-MM-DD-bugs.md`

- Created on the first `/bug` invocation of the day; subsequent bugs in the same session append to the same file
- Each bug occupies a `## Bug #<issue>: <title>` section; the issue number is back-filled immediately after the GitHub issue is filed
- The file accumulates all bugs investigated in a session and is committed at the end

---

## Per-Bug Process Flow

For each `/bug <description>` invocation:

1. **Codebase exploration** — search for affected files, functions, call sites matching the description; read relevant code to understand the context
2. **Deep investigation** — systematic-debugging Phase 1 style:
   - Trace data flow through affected components
   - Check recent git changes in relevant files
   - Identify component boundaries and interaction points
   - Document all evidence found
   - State root cause as confirmed or "best hypothesis — pending further investigation" if not fully determined
3. **Write/append spec section** — populate Description and Investigation subsections; leave Fix Design blank
4. **File GitHub issue** — title = bug title; body = spec section content (Description + Investigation); label = `bug`; back-fill issue number into spec `##` header
5. **Checkpoint** — "Bug #N recorded. Add another bug, or move to fix design?"

---

## Fix Design Phase

After the user signals "move to fix design" (all bugs captured for this session):

For **each bug** in the session, in order:

1. Present 2-3 fix approaches with trade-offs; get user approval (brainstorming-style; one question at a time if clarification needed)
2. Write approved design into the spec section under `### Fix Design`
3. Post a comment on the corresponding GitHub issue summarizing the chosen approach and linking to the spec

Then invoke **writing-plans** once. The plan covers all bugs from the session, with each task group cross-referencing its `#<issue>` number.

## Spec Section Format

```markdown
## Bug #42: <title> — [#42](https://github.com/<owner>/<repo>/issues/42)

**Filed:** YYYY-MM-DD  
**Status:** investigating | designed | planned

### Description
<user-provided description, expanded with context from investigation>

### Investigation
- **Affected files:** `src/foo.rs`, `src/bar.rs`
- **Root cause:** <confirmed cause or best hypothesis>
- **Reproduction path:** <steps or conditions that trigger the bug>
- **Evidence:** <what was found in code, traces, git log>

### Fix Design
<filled in during fix design phase; left blank until then>

### Plan Reference
<link to plan file — added after writing-plans runs>
```

## Key Constraints

- Root cause must be best-effort investigated before writing the spec entry — not just a restatement of the description
- GitHub issue is filed before fix design begins — the issue number is in the spec before the design section is written
- The plan produced by writing-plans links back to individual `#<issue>` numbers in each task group
- The skill does NOT fix code — it stops at invoking writing-plans
- `writing-plans` is the only skill invoked at the end; no other implementation skill
